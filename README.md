# worker-backend

Production-grade REST API built on **Cloudflare Workers** using Hono, Drizzle ORM, Better Auth, and D1 — with full local, staging, and production environment support.

---

## Tech Stack

| Layer | Technology |
|---|---|
| Runtime | [Cloudflare Workers](https://workers.cloudflare.com/) (`workerd`) |
| Framework | [Hono](https://hono.dev/) + [`@hono/zod-openapi`](https://github.com/honojs/middleware/tree/main/packages/zod-openapi) |
| Database | [Cloudflare D1](https://developers.cloudflare.com/d1/) (SQLite at the edge) |
| ORM | [Drizzle ORM](https://orm.drizzle.team/) |
| Auth | [Better Auth](https://www.better-auth.com/) |
| Validation | [Zod](https://zod.dev/) |
| Linting / Formatting | [Biome](https://biomejs.dev/) |
| Testing | [Vitest](https://vitest.dev/) + [`@cloudflare/vitest-pool-workers`](https://www.npmjs.com/package/@cloudflare/vitest-pool-workers) |
| Package Manager | [pnpm](https://pnpm.io/) |
| Deployment | [Wrangler](https://developers.cloudflare.com/workers/wrangler/) |

## Environment Setup

### 1. Copy and fill secrets

```bash
cp .env.example .dev.vars
```

`.dev.vars` is Wrangler's local secrets file (equivalent to `.env` for Workers). It is **gitignored**.

### 2. Generate Cloudflare binding types

```bash
pnpm types:gen
```

This runs `wrangler types` and generates `worker-configuration.d.ts` — giving you type-safe access to `env.DB`, `env.BETTER_AUTH_SECRET`, etc.

## wrangler.json — Single Config, Three Environments

All three environments are defined in one `wrangler.json` using the `env` key:

| Environment | Wrangler Env | Command |
|---|---|---|
| Local (offline) | *(base config)* | `pnpm dev:local` |
| Local (online, dev bindings) | *(base config)* | `pnpm dev` |
| Staging (Cloudflare) | `staging` | `pnpm deploy:staging` |
| Production (Cloudflare) | `production` | `pnpm deploy:prod` |

## Database Workflow

```bash
# 1. Edit schema in src/db/schema/
# 2. Generate migration
pnpm db:generate

# 3. Apply locally and verify
pnpm db:migrate:local

# 4. Deploy to staging first
pnpm db:migrate:staging

# 5. Deploy to production
pnpm db:migrate:prod
```

> ⚠️ Always apply and verify migrations in `local` → `staging` → `production` order. Never apply directly to production without staging verification.

## Logging

`pino` is **not compatible** with the Cloudflare Workers (`workerd`) runtime — it uses Node.js stream APIs.

This project uses a lightweight custom structured logger in `src/utils/logger.ts` that writes JSON to `console.*`. Cloudflare captures these logs in:

- **Local:** `wrangler tail` in your terminal
- **Staging/Production:** Cloudflare Workers Logs dashboard
- **Advanced:** Cloudflare Logpush → Axiom, Datadog, S3, etc.

```typescript
import { logger } from '@/utils'

logger.info('User created', { userId: '123' })
logger.error('DB query failed', { table: 'users', error: err.message })
```

## Key Decisions

| Decision | Reason |
|---|---|
| Single `wrangler.json` with `env` | Cloudflare-recommended pattern — no config duplication |
| Biome over ESLint + Prettier | Single tool, faster, consistent rules |
| `@hono/zod-openapi` only (no `@hono/zod-validator`) | `zod-openapi` includes validation — installing both causes conflicts |
| Custom logger over `pino` | `pino` is Node.js-only — incompatible with `workerd` |
| `@cloudflare/vitest-pool-workers` | Tests run inside real `workerd` runtime — matches production exactly |
| No `miniflare` in deps | Embedded inside Wrangler v3+ — never install separately |
| No `tsx` in devDeps | Wrangler bundles via `esbuild` internally — `tsx` is unused |
| `tests/` at root (not co-located) | Clean separation; integration tests need separate `workerd` config |
